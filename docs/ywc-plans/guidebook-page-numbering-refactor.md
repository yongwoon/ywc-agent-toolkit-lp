## Purpose

The guidebook's page order/number is currently duplicated across five independent places that must be hand-kept in sync: filename leading digits, each locale's markdown H1 text, `guidebook-nav.ts`'s hardcoded `order`/`title` fields, `scripts/generate-search-index.mjs`'s slug-discovery regex, and every inline cross-reference (prev/next footer text, README TOC/quick-jump page-number columns, the A-Z skill index in `14-skill-reference.md`). Two concrete failures already happened in one session: a deployed sidebar showed page numbers one lower than the actual files (`12/13/14` vs `13/14/15`), and a newly-added page (`16-code-structure-and-maintainability.md`) shipped without a `guidebook-nav.ts` entry and was unreachable from the sidebar despite its markdown files existing and passing `npm run test:content`. This spec makes `guidebookNavGroups`' array position the single source of truth for page order, removes numeric text from filenames going forward, and computes the displayed "N." prefix at render time — so reordering or inserting a page requires editing exactly one array, not five artifacts.

## Scope

> ⚠️ Bullet 5's file path below is SUPERSEDED by Iteration 1 — see [Iteration 1 Amendments](#iteration-1-amendments). All other bullets are unaffected.

- Add a `displayNumber` (or equivalent) computed value to the guidebook nav data model, derived from a page's 1-based index in the flattened `guidebookPages` array (global numbering, matching the current 01–16 scheme — not reset per group), and thread it into the three render sites that currently print `page.title` raw: `sidebar-nav.tsx`, `prev-next-nav.tsx`, and the breadcrumb / `<h1>` in `page.tsx`.
- Strip the `"NN. "` prefix from `guidebookNavGroups`' `title` field (16 entries) in `guidebook-nav.ts` — title becomes plain descriptive text, the number is no longer stored there.
- Remove the `order` field from `GuidebookPageMeta` and from every entry in `guidebookNavGroups` (16 entries) — array position becomes authoritative; `inferOrderFromFilePath` in `guidebook-content.ts` becomes dead code and is deleted.
- Rename future-added pages' files without a leading digit (descriptive slug only) going forward. **Existing 16 files × 5 locales (80 files) are NOT renamed in this pass** — see Out of Scope.
- Update `scripts/generate-search-index.mjs`'s `readSlugs()` regex (`/^\d+-.+\.md$/`) to also accept non-numeric slugs, since new pages will not carry leading digits.
- Update `docs/guides/guidebook/README.md`-derived quick-jump / TOC tables and `14-skill-reference.md`'s A-Z index generation approach (the `build-az-index.mjs`/`insert-az-index.mjs` scratch scripts from this session) to compute page-number columns from `guidebookNavGroups` position rather than hand-typing `[16](...)`-style literals, for the pages this spec touches going forward.
- A machine-checkable invariant (script or test) asserting `guidebookNavGroups` slugs exactly match the set of markdown files present in `src/content/guidebook/en/` (catches the "page 16 unreachable" failure class at build/CI time instead of silently shipping).

## Out of Scope

- **Renaming the existing 80 markdown files (16 pages × 5 locales) to strip their leading digits.** The advisor's recommended variant explicitly avoids this: H1 text and filenames for the 16 already-shipped pages keep their current numeric prefix as-is; only the *rendering* and *nav-array* layers stop trusting that prefix as the order source. Retrofitting the existing files is a separate, lower-priority follow-up once the render-time mechanism is proven — bundling it here would multiply this spec's blast radius (80 file edits + redirect infrastructure) for a purely cosmetic content-consistency win.
- **Redirect infrastructure for renamed URLs.** Because existing files are not renamed in this pass, no slugs change and no GitHub Pages redirect stubs are needed *yet*. The `RootRedirectPage` pattern (`src/app/(root)/page.tsx`) is documented here as the precedent to follow *if and when* a future pass renames existing slugs, but building that infrastructure now — before any slug actually changes — would be speculative.
- **Rewriting every inline cross-reference inside the 80 existing markdown files** (prev/next footer text like `[다음: 16. ...]`, README quick-jump `[16](...)` columns, the A-Z index's per-skill page links). These stay as literal numbered text for the existing 16 pages; only the generation *scripts* for future content (the A-Z index builder, README TOC row builder) are updated to compute numbers instead of hand-typing them, per Scope.
- **Renumbering the actual page sequence** (e.g. moving "Prerequisites" earlier). This spec changes *how* order is stored/rendered, not the current order itself.
- **The `develop-with-llm/docs/guides/guidebook/` mirror** and the `codex-skill` bundle — both are independently maintained per this project's and `develop-with-llm`'s own CLAUDE.md conventions (no auto-sync); out of scope by that existing convention, not a new decision made here.
- **Group-relative numbering** (e.g., resetting to "1." at the start of each sidebar group). The current scheme is global (01→16 across all groups) and this spec preserves that; switching to group-relative numbering is a product decision, not a plumbing fix, and is not requested.

## Architecture Decision

Framed to `ywc-architect` (bounded payload: this duplication problem, the existing fallback-chain code in `guidebook-content.ts`, and the `generateStaticParams` dependency on `guidebookNavGroups`):

- **Option A** (per-file `order:` frontmatter authoritative) was rejected: it introduces a *second* independent ordering authority that must stay in lockstep with `guidebookNavGroups`' own membership/grouping (which is unavoidably load-bearing for `generateStaticParams`, `dynamicParams = false`) — trading one drift vector for two.
- **Option B** (array position authoritative) was accepted, with the advisor's suggested lower-risk variant: instead of bulk-rewriting all 80 H1s to strip their numeric prefix in the same pass, leave existing H1 text as-is for now and have the render layer *override* the displayed number using the array index, never trusting the H1's literal digits. This reaches the same single-source-of-truth outcome with a much smaller migration surface (`guidebook-nav.ts` + 3 render call sites + 1 script regex, not 80 content files).
- Type-shape consequence: `guidebookNavGroups` entries lose `order`; `title` loses its numeric prefix (becomes plain text, e.g. `"Managing Code Structure and Maintainability"` not `"16. Managing Code Structure and Maintainability"`). `normalizeGuidebookFrontmatter` in `guidebook-content.ts` stops resolving `order` at all (that field is deleted from `GuidebookFrontmatter`); `inferOrderFromFilePath` becomes dead code and is deleted. `inferTitleFromFilePath` survives only as the last-resort fallback when an H1 is missing.

## Acceptance Criteria

> ⚠️ AC6 below is SUPERSEDED by Iteration 1 — see [Iteration 1 Amendments](#iteration-1-amendments). AC1–AC5 are unaffected.

- [ ] **AC1 — Reordering requires exactly one file edit.** When a maintainer moves a page to a different position by editing only the array order in `guidebookNavGroups` (no other file touched), the system re-renders the sidebar, prev/next nav, and page `<h1>` with the new numbers reflecting the new array position, observable as: after `npm run build`, the generated static HTML for the moved page and its new neighbors shows the updated `"NN. "` prefix with no other file having been edited in the commit.
- [ ] **AC2 — Inserting a page without registering it in `guidebookNavGroups` fails the build (not silently ships).** When a new markdown file exists under `src/content/guidebook/en/<slug>.md` but has no corresponding entry in `guidebookNavGroups`, the invariant check (Scope's machine-checkable invariant) fails with a non-zero exit code and a message naming the orphan slug, observable as: running the invariant script/test locally against a deliberately-unregistered fixture file exits non-zero and prints the fixture's slug.
- [ ] **AC3 — Sidebar numbers match the current global 01–16 scheme exactly, for all 5 locales, before and after this refactor.** For each locale, the sidebar's rendered `"NN. "` prefixes read 01 through 16 in the same page order as today, observable as: a diff of the sidebar's rendered `page.title`-adjacent number text before vs. after this change is empty for all 5 locales (`ko/en/ja/zh/es`) — this is a rendering-mechanism change, not an ordering change.
- [ ] **AC4 — `order` is fully removed from the type surface, not merely unused.** After this change, `GuidebookPageMeta` and `GuidebookFrontmatter` no longer declare an `order` field, observable as: `npx tsc --noEmit` reports no errors, and `grep -n "order" src/components/guidebook/guidebook-nav.ts src/lib/guidebook-content.ts` returns zero matches outside of unrelated identifiers (e.g. a variable named `orderBy` in unrelated code, if any — verified not to exist by the complement-check row above).
- [ ] **AC5 — The search index is not silently missing any page.** After `scripts/generate-search-index.mjs`'s regex is broadened, running `npm run build` (which triggers `prebuild` → `generate-search-index.mjs`) produces exactly 16 entries per locale in each `src/data/guidebook-search.<locale>.json`, matching the count before this change, observable as: `jq 'length' src/data/guidebook-search.en.json` prints `16` both before and after.
- [ ] **AC6 — A future non-numeric-filename page is discoverable end-to-end.** A test fixture page (`src/content/guidebook/en/fixture-numbering-test.md` + matching entry in `guidebookNavGroups`, removed after verification) with no leading digit in its filename appears in the sidebar with a correctly-computed number, is included in the search index, and is reachable via `generateStaticParams`, observable as: `npm run build` succeeds, the fixture slug appears in `src/data/guidebook-search.en.json`, and the fixture's static HTML output exists under the build output directory at the fixture's slug path.

## Functional Requirements

> ⚠️ FR-1, FR-3, FR-4, FR-5 below are SUPERSEDED by Iteration 1 — see [Iteration 1 Amendments](#iteration-1-amendments). FR-2 is unaffected and remains authoritative as written.

### FR-1: Remove `order` as a stored field; compute display number from array position (superseded)

`GuidebookPageMeta.order` (in `guidebook-nav.ts`) and `GuidebookFrontmatter.order` (in `guidebook-content.ts`) are deleted. `guidebook-nav-content.ts`'s `loadLocalizedGuidebookNav` computes a `displayNumber` per page as `1 + guidebookPages.findIndex((p) => p.slug === page.slug)` (or an equivalent single pass building an index map once, to avoid O(n²) — implementation's choice, not prescribed here) after building the per-locale `pages` array, and includes it on each returned page object.

### FR-2: Strip numeric prefix from `guidebookNavGroups`' `title` field

All 16 `title` string literals in `guidebook-nav.ts` (e.g. `"01. Introduction"` → `"Introduction"`) are edited to remove the `"NN. "` prefix. This field remains the English fallback used only when a locale's content file fails to load (per the existing fallback chain in `guidebook-nav-content.ts:32-36`); it is not the primary source of the displayed title in normal operation.

### FR-3: Render sites prefix the computed number, not stored text

`sidebar-nav.tsx`, `prev-next-nav.tsx`, and `page.tsx` (the `<h1>` and — per Edge Case below — optionally `generateMetadata`'s SEO title) format the displayed title as `${displayNumber}. ${title}` (or the locale-appropriate equivalent punctuation, if any locale's existing convention differs — verified in Edge Cases that all 5 locales currently use the same `"NN. "` ASCII-period-space convention, so no per-locale formatting branch is needed).

### FR-4: Search-index slug discovery accepts non-numeric filenames

`scripts/generate-search-index.mjs`'s `readSlugs()` regex is broadened from `/^\d+-.+\.md$/` to a pattern accepting both the existing numeric-prefixed filenames and future non-numeric slugs, without accidentally matching non-content files (e.g. a hypothetical `.gitkeep` or a partial/draft file using a different extension — the existing `.md`-only constraint is preserved).

### FR-5: Build-time invariant catches orphan/missing registrations

A script or test (placed under `scripts/` or `test/`, following the existing `test/check-guidebook-paths.mjs` precedent's location convention) asserts that the slug set in `guidebookNavGroups` exactly equals the slug set of `.md` files present in `src/content/guidebook/en/` (bidirectional: no array entry without a file, no file without an array entry). This is wired into `npm run test:content` or an equivalent existing CI-invoked script, not a new standalone command a developer must remember to run manually.

## Edge Cases

- **A locale's content file is missing or fails to load entirely** (the existing `content === undefined` branch in `page.tsx`). `displayNumber` is still computable (it depends only on `guidebookNavGroups` array position, not on content load success), so the numbered fallback title (`guidebook-nav.ts`'s English `title`, now unprefixed) still renders correctly with its number — no regression from today's behavior, which already handles this case for the title text itself.
- **SEO `<title>` tag numbering.** `generateMetadata` currently uses `navPage.title` (today already numbered, from the per-locale-content-derived title) for the HTML `<title>`/OpenGraph tags. Whether the numbered prefix belongs in the SEO title is a content-strategy question this spec does not resolve — **implementation should default to keeping the number in the SEO title** (matching current behavior exactly, zero visible change to search-engine-facing metadata) and treat removing it as a separate, explicitly-requested follow-up if desired.
- **The invariant check (FR-5) during the transition period.** Since existing 80 files keep numeric filenames (Out of Scope) while future files will not, the invariant must not assume a filename pattern — it compares the *slug* (derived from filename minus extension, whatever its shape) against `guidebookNavGroups`' `slug` field, which already works for both numeric and non-numeric slugs without modification.
- **Locale-specific numbering punctuation.** Verified via the existing rendered files (this session's own edits to `14-skill-reference.md`, `16-code-structure-and-maintainability.md`, etc. across all 5 locales) that every locale uses the identical ASCII `"NN. "` convention (period + single space) — Japanese does not use a full-width `。` for this, Chinese does not use `、`. FR-3 assumes this holds; if a future locale addition uses different punctuation, that locale's render path needs a small branch, not a redesign.
- **`guidebookPages` array is `as const` (readonly).** `findIndex` and array methods work fine on `readonly` arrays; no mutation is needed or introduced by FR-1's computation.

## Non-Functional Requirements

- **NFR-1 (from architecture advisor).** This pass introduces zero URL/slug changes (Out of Scope), so no redirect infrastructure is required now. **If a future pass renames any of the existing 80 files' slugs**, that follow-up spec must ship a static redirect stub per renamed slug × locale, reusing the verified `src/app/(root)/page.tsx` mechanism (client-side `useEffect` + `navigator`-independent static fallback links — not a meta-refresh tag, correcting the initial assumption), driven by an explicit, repo-checked-in old-slug → new-slug map that is itself enumerated through `generateStaticParams`, so no previously indexed or bookmarked `/guidebook/<old-slug>/` URL 404s after the GitHub Pages static export redeploys (no server-side redirects are available on this static host).
- **NFR-2.** `npm run build`'s bundle-budget check (`test/build-verification/check-bundle-budget.mjs`, part of the `Build & Bundle Budget` CI job) must continue to pass — the `displayNumber` computation is O(pages) and adds no meaningful bundle weight, but implementation should confirm the CI job stays green rather than assume it.
- **NFR-3.** No change to `dynamicParams = false` / static-export behavior — this spec does not touch routing, only display-text computation and the search-index script.

## Dependencies

- Builds on PR #27 (`docs/guidebook-az-index-and-mission-fix`) having merged first — that PR is what added page 16 and is the concrete example motivating this refactor; implementing against an unmerged base would conflict.

## Open Questions

- Should the eventual "rename the existing 80 files" follow-up (Out of Scope here) happen at all, or is the render-time-override variant sufficient permanently? The advisor's variant was chosen specifically to make this optional — recommend deferring the decision until this pass has been live for a while and the actual pain (if any) of numeric-prefixed existing filenames is felt, rather than deciding now.
- Should `displayNumber` reset per sidebar group (matching many doc-site conventions of "Chapter 1, Section 1.1") or stay global (current behavior, preserved by this spec per Out of Scope)? Recommend keeping global for this pass since changing the *scheme* (not just its storage mechanism) is a product decision outside this refactor's purpose.
- **(New, Iteration 1)** Should FR-4's search-index consistency gap (Edge Case, Iteration 1) be closed in a follow-up spec proactively, or only if/when a non-numeric page is actually added and someone notices the search-result title lacks a number? Recommend waiting — AC6's fixture is removed after verification, so no real page hits this gap until the first genuine non-numeric page ships, which is itself deferred (Out of Scope: "existing 80 files are not renamed in this pass").
